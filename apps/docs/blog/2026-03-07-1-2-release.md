`1.2` focuses on stabilizing drag behavior across packages by standardizing on `@wonderlandlabs-pixi-ux/observe-drag`.
The former drag package is being replaced by a `@wonderlandlabs-pixi-ux/observe-drag`
which uses straight to event hooks for a more direct drag hook. 

There are still outstanding questions - the resizer component now needs to call `app.render()` for reasons that are 
unclear even though not changes in teh render tree are being made, and made inside the ticker loop.  

## Highlights

- Added `dragObserverFactory` in `@wonderlandlabs-pixi-ux/observe-drag` with:
  - per-stage lock scope by default
  - optional shared `activePointer$` injection at factory creation
  - configurable inactivity failsafe (`abortTime`, `0` disables)
- Migrated window dragging to observer-based drag subscriptions coordinated through `WindowsManager`.
- Kept compatibility aliases where needed (`observeDrag`, `dragTargetDecorator`) while moving new usage toward `dragObserverFactory` + `dragDecorator`.

## Deprecations

- `@wonderlandlabs-pixi-ux/drag` is now **deprecated** for `1.2.x`.
  - Migration target: `@wonderlandlabs-pixi-ux/observe-drag`.
- `trackDrag` is no longer exported from `@wonderlandlabs-pixi-ux/resizer`.
  - Legacy implementation remains under `_deprecated` internally, but public API consumers should migrate to observer-based drag flows.

## Migration Direction

If your sole concern is to allow one layer to be dragged as you move you can establish automatic
dragging through use of the dragDecorator that will automatically shift the target along as you drag.

Prefer:

```ts
import dragObserverFactory, {dragDecorator} from '@wonderlandlabs-pixi-ux/observe-drag';

const observeDown = dragObserverFactory({stage: app.stage});
const subscription = observeDown(target, dragDecorator(), {
  dragTarget: container,
  abortTime: 1000,
});
```

Over legacy `DragStore` or `trackDrag` entrypoint usage.

 More advanced side effects can be managed with custom hooks: `{ onStart, onMove, onEnd }`
